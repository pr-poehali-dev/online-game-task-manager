## 1. Фикс раздела «К рестарту»

### Что было не так
В разделе «К рестарту» задачи группируются по серверам. Список серверов для группировки
был жёстко задан (3 конкретных сервера). Если у задачи стоял сервер, которого нет в этом
списке (например, изменённый/дополнительный сервер в вашей команде), задача пропадала из
раздела визуально — при этом счётчик задач наверху продолжал считать её корректно.
Из-за этого выглядело так, будто «счётчик есть, а список пуст».

### Что исправлено
Теперь для серверов, не входящих в основной список, автоматически создаётся отдельный
блок группировки — ни одна задача больше не потеряется из вида.

### Дополнительный фикс: карточка кликабельна не полностью
В том же разделе «К рестарту» карточка задачи открывалась только по клику на заголовок —
клик по остальной части карточки (бейджи, статус деплоя, исполнители, сервер) не работал.
Теперь кликабельна вся карточка целиком, кнопки «Готово» / «В архив» по-прежнему работают
как обычно и не открывают карточку.

## 3. Фикс удаления задачи с доски (не возвращается больше)

### Что было не так
При нажатии «Удалить» на карточке задачи (полное удаление, не архивирование) задача
пропадала с экрана, но при определённых условиях не удалялась из базы данных и
возвращалась после обновления страницы — независимо от того, сколько ждать.

Настоящая причина — в базе данных: у каждой задачи, которой хоть раз назначали
исполнителя, есть служебная запись в таблице статистики `task_assignment_events`.
Эта запись была жёстко привязана к задаче (внешний ключ) без разрешения на
автоматическое удаление. Из-за этого база данных **отклоняла** запрос на удаление
такой задачи, backend не обрабатывал эту ошибку и молча возвращал «успех», а
фронтенд не проверял реальный результат запроса — поэтому задача просто
возвращалась после перезагрузки.

Дополнительно поправлена задержка в 5 секунд перед отправкой запроса на удаление
(теперь запрос уходит сразу, кнопка «Восстановить» в уведомлении по-прежнему
работает — при нажатии задача создаётся заново).

### Что исправлено
- Убрана жёсткая связь в базе данных, мешавшая удалению (миграция).
- Backend теперь дополнительно удаляет связанные служебные записи и корректно
  сообщает об ошибке, если удаление не удалось.
- Фронтенд проверяет результат запроса на удаление и восстанавливает задачу на
  экране с понятным сообщением об ошибке, если удаление не прошло.

### Файлы для переноса
```
db_migrations/V0035__drop_task_assignment_events_fkey.sql   — накатить на БД
backend/tasks/index.py                                       — заменить целиком

```

### Применение миграции
```bash
cd /var/www/era/deploy

MAIN_DB_SCHEMA="ВАША_СХЕМА" \
bash apply_migrations.sh
```
(та же команда, что и при обычном обновлении — она применяет только новые файлы миграций)

После замены `backend/tasks/index.py` перезапустите backend:
```bash
sudo systemctl restart era-backend
```

## 4. Фикс выравнивания в окне выбора статуса деплоя

### Что было не так
При переносе карточки задачи между колонками доски (например из «Done» в «In Progress»)
появляется окно выбора статуса деплоя. Если текст варианта не помещался в одну строку
(например «На тестировании (залито на тестовый)»), иконка слева выравнивалась по центру
всего пункта, а не по верхней строке текста — из-за этого иконка и текст выглядели
смещёнными друг относительно друга.

### Что исправлено
Иконка теперь всегда закреплена у верхней строки текста, а сам текст выровнен по левому
краю с аккуратным межстрочным интервалом — выглядит опрятно независимо от того,
переносится текст варианта на одну или две строки.

### Файл для переноса
```

```

## Как перенести

Проще всего — скопировать перечисленные файлы из этой папки поверх файлов на сервере,
сохранив пути (`index.html` в корень, остальное — в `public/`).

Либо применить патч (только для текстовых изменений в `index.html` и `Restart.tsx`,
картинки патчем не переносятся — их нужно скопировать отдельно, они уже лежат готовыми
в папке `public/` рядом с этим README):

```bash
cd /var/www/era   # корень вашего проекта
git apply update/changes.patch
# либо, если это не git-репозиторий:
patch -p1 < update/changes.patch
```

## После переноса
```bash
npm run build
```
Backend перезапускать нужно только из-за пункта 3 (см. выше).

- Откройте вкладку «К рестарту» — задача с любым сервером теперь отображается в своём блоке.
- Обновите вкладку браузера — должна появиться новая иконка (может закэшироваться, при
  необходимости очистите кэш браузера).
- Отправьте ссылку на сайт себе в Telegram — должно подтянуться новое превью с логотипом.
- Назначьте исполнителя на тестовую задачу, удалите её с доски и обновите страницу —
  задача не должна вернуться.
- Перетащите карточку задачи из «Done» в «In Progress» — в окне выбора статуса деплоя
  иконка и текст варианта «На тестировании (залито на тестовый)» должны быть аккуратно
  выровнены.
- Откройте FAQ в личном кабинете — должно быть 21 вопрос, включая новые про вложения,
  приватные статьи, журнал активности и статистику участника.

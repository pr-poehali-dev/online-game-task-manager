## 3. Автоматические напоминания о приближающемся дедлайне

### Что добавлено
Новая облачная функция `backend/deadline-reminders` проверяет все незавершённые
задачи с дедлайном и рассылает напоминания автору и исполнителям:
- за **24 часа** до срока
- за **6 часов** до срока
- за **30 минут** до срока

Напоминание приходит как уведомление в приложении (колокольчик), а тем, кто входил
через Telegram-бота — ещё и сообщением в Telegram со ссылкой на задачу. Каждое
напоминание отправляется только один раз (отслеживается в базе), повторов не будет.

**Важно:** платформа не запускает облачные функции по расписанию сама — функцию
нужно "будить" регулярно через бесплатный внешний cron-сервис. Подробная пошаговая
инструкция — в файле `CRON_SETUP.md` в этой же папке.

Функция защищена секретным заголовком `X-Cron-Secret` — значение должно совпадать
с секретом `CRON_SECRET`, заданным в проекте. Без верного заголовка функция отвечает
`403 Forbidden` и ничего не отправляет.

### Файлы для переноса
```
db_migrations/V0039__add_deadline_reminders_sent.sql   — накатить на БД

backend/deadline-reminders/                             — новая папка целиком (index.py, requirements.txt, tests.json)
```

### Переменная окружения
Убедитесь, что в окружении backend задан секрет `CRON_SECRET` (любая длинная
случайная строка) — без него функция будет работать без защиты (403 не сработает,
любой знающий адрес сможет её вызвать).

## 4. Превью текста в уведомлениях (первые 100 символов)

### Что было не так
Уведомления о назначении задачи, новом комментарии, ответе на комментарий и
упоминании показывали только название задачи/идеи — без единого слова из самого
текста. Чтобы понять, о чём речь, приходилось открывать задачу.

Из описания задачи (оно хранится в формате rich text) HTML-теги перед показом
убираются — остаётся только читаемый текст. Если текст длиннее 100 символов, в
конце добавляется многоточие.

В списке уведомлений (колокольчик) текст теперь может занимать до двух строк

3 текстовых журнала (по одному на каждый сервер: С4х1, HFx3 old, HF new),

Есть кнопка «Копировать» — забирает весь текст журнала в буфer обмена одним кликом
(удобно, чтобы вставить в чат команды или отправить игрокам).

- Новая облачная функция `backend/patchnotes` — отдаёт записи журнала (только чтение,

- `backend/tasks/index.py` — при архивации задачи с исходом `done` из колонки `restart`

## 11. Фикс: иногда не удавалось загрузить файл (лимит Nginx конфликтовал с лимитом приложения)

### Что было не так
На self-hosted сервере (Nginx + MinIO) был установлен лимит `client_max_body_size 20M`
на размер тела запроса — а приложение при этом разрешает вложения размером до 300 МБ
(`MAX_FILE_SIZE` в backend). Файл при загрузке кодируется в base64, что увеличивает его
размер примерно на треть — то есть уже файл размером ~15 МБ после кодирования упирался
в лимит Nginx и обрывался ДО того, как запрос вообще доходил до кода приложения.
Именно поэтому загрузка срабатывала «иногда» — маленькие файлы проходили, а более
крупные обрывались с невнятной ошибкой «Не удалось загрузить файл», даже если по
логике приложения (до 300 МБ) файл был вполне допустимым.

Таймаут backend (60 секунд) тоже был занижен — декодирование большого файла и его
отправка в MinIO может занимать больше времени.

### Что исправлено
- Лимит `client_max_body_size` для маршрута `/api/` увеличен до 420 МБ (с запасом на
  base64-накладные расходы к 300 МБ реального файла), обычные страницы сайта
  по-прежнему ограничены 20 МБ.
- Таймауты Nginx (`proxy_read_timeout`, `proxy_send_timeout`) и backend (gunicorn,
  включая Docker-вариант) увеличены с 60 до 180 секунд.
- Backend теперь корректно обрабатывает повреждённые/оборванные данные при загрузке
  (например, если соединение прервалось на середине) — вместо голой ошибки сервера
  возвращает понятную причину, а фронтенд показывает сообщение «Файл повреждён при
  загрузке — проверьте соединение и попробуйте ещё раз».

## 12. Ссылки в комментариях и описаниях становятся кликабельными автоматически

Если в тексте комментария к задаче или идее написать адрес вида `http://...` или
`https://...`, он теперь автоматически превращается в кликабельную ссылку, которая
открывается в новой вкладке — руками оборачивать в разметку не нужно.

То же самое работает и в описании задачи/идеи (поле с форматированием) — при вводе
или вставке текста со ссылкой она сразу становится кликабельной прямо во время
редактирования.

### Файлы для переноса
```
src/components/RichEditor.tsx
src/pages/index/TaskModalShared.tsx   — (уже входит в п.6, если переносите его)
src/pages/index/ideas/shared.tsx      — (уже входит в п.6, если переносите его)
```

### Проверка
- Вставьте в комментарий к задаче ссылку вида `https://example.com` — она должна
  сразу подсветиться как кликабельная ссылка после отправки комментария.
- В описании задачи (поле с форматированием) наберите или вставьте ссылку — она
  должна стать кликабельной прямо в процессе набора текста.

## Итоговая инструкция по переносу всех изменений

Если переносите всё сразу (рекомендуется — большинство файлов пересекаются между
пунктами, поэтому переносить их по отдельности дольше и легче что-то упустить):

### 1. Скопируйте файлы
Скопируйте содержимое папок `backend/`, `src/`, `deploy/` из этого обновления
поверх соответствующих папок на сервере, сохраняя пути. Список новых backend-функций
(нужно перенести целиком, включая `requirements.txt` и `tests.json`):
- `backend/deadline-reminders/`
- `backend/patchnotes/`

### 2. Накатите миграции БД по порядку
```bash
cd /var/www/era/deploy
DATABASE_URL="postgresql://era_user:ВАШ_ПАРОЛЬ@localhost:5432/era_db" \
MAIN_DB_SCHEMA="ВАША_СХЕМА" \
bash apply_migrations.sh
```
Скрипт применяет только новые файлы миграций по порядку номеров (V0037…V0048)
автоматически. Прочитайте примечание к пункту 13 про миграцию V0046 — она содержит
ручное исправление данных конкретной тестовой задачи и может быть неприменима
к вашей базе (не критично, просто не даст эффекта).

### 3. Обновите конфигурацию сервера (пункт 11 — фикс загрузки файлов)
Обязательно выполните шаги из `UPLOAD_FIX_SETUP.md` — без этого лимит Nginx
на размер запроса останется старым, и фикс из пункта 11 не заработает.

### 4. Соберите фронтенд
```bash
cd /var/www/era
npm run build
```

### 5. Перезапустите backend
```bash
sudo systemctl restart era-backend
```
(Если используете Docker — см. отдельные команды пересборки в `UPLOAD_FIX_SETUP.md`.)

### 6. Настройте внешний cron (пункт 3 — напоминания о дедлайне)
Следуйте инструкции в `CRON_SETUP.md` — без этого шага напоминания о дедлайне
не будут отправляться (функция не запускается сама по расписанию).

### 7. Проверьте секреты
Убедитесь, что в окружении backend задан `CRON_SECRET` (см. пункт 3).

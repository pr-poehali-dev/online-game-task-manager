# Обновления от 21 июля

Один новый раздел — «Патчи» (полностью новый функционал: файловое дерево
клиентского патча по каждому серверу) и его доработки в течение дня, плюс
статистика места на диске VPS во вкладке «Файлы» админки и исправление
навигации по постоянным ссылкам (`/task/:id`, `/idea/:id`, `/kb/:id`).

Краткое содержание:
1. Новый раздел «Патчи» — файловое дерево клиентского патча по серверам
2. Потоковая (chunked) загрузка файлов — обход лимита ~3 МБ на один запрос
3. Привязка/открепление уже загруженного файла к задаче прямо в дереве
4. Пользовательские корневые папки — создание и удаление (только пустых)
5. Кликабельный логотип «ЭРА» — быстрый переход на доску
6. Исправление: адрес в браузере не возвращался на доску при переходе из карточки
7. Статистика места на диске VPS во вкладке «Файлы» в админке
8. 3 новых вопроса в FAQ личного кабинета — про «Патчи» и статистику диска

## 1. Новый раздел «Патчи»

### Что это
Раздел показывает единое дерево файлов клиентского патча — отдельное
для каждого из трёх серверов (С4х1, HFx3 old, HF new). Дерево организовано
по 9 фиксированным корневым папкам, которые соответствуют структуре клиента
игры: `animations`, `data`, `l2text`, `maps`, `staticmeshes`, `System`,
`System_eng`, `systextures`, `textures` — плюс пользователь может добавлять
свои собственные корневые папки (см. пункт 4).

Кто угодно из авторизованных участников может открыть раздел, посмотреть
дерево и скачать любой файл. Загружать, удалять файлы, привязывать их
к задачам и управлять папками могут только администраторы и участники
с правом полного редактирования задач (`task_edit_own`).

### Как загружать файлы
1. Откройте раздел «Патчи» в верхнем меню, выберите нужный сервер.
2. При необходимости выберите задачу в выпадающем списке над деревом —
   тогда все файлы, загруженные ниже, автоматически прикрепятся к ней.
3. Перетащите папку (например «System» или «data») прямо на нужную
   корневую папку в дереве — вложенная структура сохранится один в один.
4. Во время загрузки виден прогресс-бар с процентом и номером текущего
   файла из очереди, загрузку можно отменить кнопкой «Отменить».

### Скачивание файлов задачи
Если в выпадающем списке выбрана задача, рядом появляется кнопка
«Скачать файлы задачи (N)» — она собирает все прикреплённые к этой
задаче файлы в один zip-архив.

### Файлы для переноса
```
backend/patches/index.py

src/pages/index/Patches.tsx
src/pages/index/PatchesTreeFolder.tsx
src/pages/index/patchesApi.ts

src/pages/index/IndexMain.tsx
src/pages/index/IndexTopbar.tsx

```

Обратите внимание: логика раздела разбита на 4 файла вместо одного большого
`Patches.tsx` — `patchesUtils.ts` (чистые утилиты и типы: дерево, форматирование
размера, сбор файлов из перетащенной папки), `patchesApi.ts` (сетевой слой,
загрузка кусочками), `PatchesTreeFolder.tsx` (рекурсивный компонент дерева) и
сам `Patches.tsx` (состояние страницы и разметка). При переносе нужны все 4 файла.

### Проверка
- Откройте раздел «Патчи», переключите сервер — дерево должно
  перезагрузиться, показав 9 фиксированных корневых папок.
- Перетащите тестовую папку на любую корневую — должен появиться
  прогресс-бар, файлы должны появиться в дереве с правильной вложенностью.
- Выберите задачу в списке, скачайте архив файлов задачи — должен
  скачаться zip с правильными файлами.

## 2. Потоковая (chunked) загрузка — обход лимита на размер запроса

### Что было не так
Первая версия раздела отправляла содержимое файла одним запросом целиком
(base64 в теле JSON). Инфраструктура физически режет тело одного запроса
к серверной функции на уровне ~3 МБ — большинство настоящих файлов
клиентского патча (текстуры, модели, `.u`/`.dll` файлы) вообще не могли
загрузиться: запрос либо отклонялся с ошибкой 413, либо функция падала
без ответа при чуть большем объёме.

### Что исправлено
Каждый файл теперь автоматически режется на стороне браузера на кусочки
по 1.5 МБ и отправляется последовательно отдельными запросами:
1. `file_init` — сообщает серверу путь файла и (если выбрана) задачу,
   получает временный `fileId`.
2. `file_chunk` — грузит по очереди каждый кусочек, кусочки временно
   складываются в S3.
3. `file_complete` — сервер склеивает все кусочки в один файл (до 200 МБ),
   сохраняет его в постоянном месте и записывает в базу.
4. `file_abort` — если загрузка отменена или произошла ошибка, все
   временные кусочки этого файла удаляются из S3.

Проверено на файле 30 МБ — загружается без ошибок с прогресс-баром и
возможностью отмены.

## 3. Привязка/открепление уже загруженного файла к задаче

### Что было не так
Прикрепить файл к задаче можно было только в момент его загрузки — чтобы
связать уже лежащий в дереве файл с новой задачей, пришлось бы заливать
его заново.

### Что исправлено
Выберите задачу в выпадающем списке над деревом, наведите курсор на нужный
файл — справа появится иконка скрепки. Клик прикрепляет файл к выбранной
задаче, повторный клик — открепляет, без повторной загрузки самого файла.
Один файл может быть одновременно прикреплён к нескольким задачам —
привязки не заменяют, а дополняют друг друга.

На backend за это отвечает действие `toggle_task`: по пути файла и id
задачи оно добавляет или убирает задачу из списка привязанных, не трогая
сам файл и его содержимое.

⚠️ Важный нюанс: если файл по одному и тому же пути загружается повторно
(новая версия поверх старой), в дереве остаётся только последняя версия —
сервер не хранит историю версий. При этом все прежние привязки к задачам
за файлом сохраняются, плюс добавляется новая.

## 4. Пользовательские корневые папки

### Что добавлено
Рядом с названием сервера в разделе «Патчи» появилась кнопка «+» —
по клику открывается поле для ввода названия новой корневой папки (только
латиница, цифры, «_» и «-»). Новая папка сразу появляется в общем списке
рядом с системными и в неё точно так же можно перетаскивать файлы.

У каждой **пользовательской** папки при наведении появляется иконка
корзины — но удалить получится только если папка пустая (защита от
случайной потери данных через проверку на backend). Системные папки
(9 фиксированных) удалить нельзя.

Backend-действия: `add_root` (создание, с проверкой имени и что оно не
совпадает с системным корнем) и `delete_root` (удаление, с проверкой что
в папке нет ни одного файла — иначе возвращается ошибка `root_not_empty`).
Список пользовательских папок хранится в отдельной таблице
`patch_custom_roots` (миграция V0052).

## 6. Исправление: адрес в браузере не возвращался на доску

### Что было не так
Если пользователь находился на постоянной ссылке карточки (`/task/:id`,
`/idea/:id`, `/kb/:id`) и нажимал «Доска» (или логотип, или любой другой
раздел меню), отображение переключалось на нужный раздел, но адрес
в браузере оставался прежним (например `/idea/3`). При обновлении страницы
(F5) это приводило к тому, что снова открывалась та же карточка вместо
выбранного раздела.

### Что исправлено
Переключение раздела через меню теперь дополнительно возвращает адрес
браузера на корень (`/`), если пользователь был на постоянной ссылке
карточки — реализовано через обёртку `changeView` в `src/pages/Index.tsx`,
которая используется вместо прямого `setView` при передаче в сайдбар,
топбар и главный контейнер.

### Файлы для переноса
```
src/pages/Index.tsx
src/pages/index/useDeepLinks.ts
```

### Проверка
- Откройте любую идею или задачу по прямой ссылке, нажмите «Доска» —
  адрес должен смениться на `/`. Обновите страницу — должна открыться
  доска, а не та же карточка.

## 7. Статистика места на диске VPS во вкладке «Файлы»

### Что добавлено
Поскольку backend разворачивается как обычный Python-процесс прямо на
вашем VPS (не облачная функция), теперь во вкладке «Файлы» в админке
(`Ядро → Файлы`) над списком вложений показывается реальная статистика
диска сервера: сколько занято, сколько всего, сколько свободно — в виде
полосы загрузки с цветовой индикацией (жёлтая при 75%+ заполнении, красная
при 90%+).

Backend читает эти данные через `shutil.disk_usage()` по пути, который
можно настроить переменной окружения `DISK_USAGE_PATH` (по умолчанию
корень `/`).

### Файлы для переноса
```
backend/admin/index.py
src/pages/admin/FilesModal.tsx
src/pages/admin/adminShared.ts
```

### Проверка
- Откройте `Ядро → Файлы` — над списком файлов должна быть полоса
  с текстом «Занято X ГБ из Y ГБ · свободно Z ГБ», соответствующая
  реальному месту на вашем сервере.

## Как перенести все изменения к себе на сервер

### Шаг 1. Скопируйте новые и обновлённые файлы

```bash
cd /var/www/era   # корень вашего проекта на сервере

mkdir -p backend/patches
cp update/2026-07-21/backend/patches/index.py backend/patches/index.py
cp update/2026-07-21/backend/patches/requirements.txt backend/patches/requirements.txt
cp update/2026-07-21/backend/patches/tests.json backend/patches/tests.json

cp update/2026-07-21/backend/admin/index.py backend/admin/index.py

cp update/2026-07-21/src/pages/index/Patches.tsx src/pages/index/Patches.tsx
cp update/2026-07-21/src/pages/index/PatchesTreeFolder.tsx src/pages/index/PatchesTreeFolder.tsx
cp update/2026-07-21/src/pages/index/patchesApi.ts src/pages/index/patchesApi.ts
cp update/2026-07-21/src/pages/index/patchesUtils.ts src/pages/index/patchesUtils.ts
cp update/2026-07-21/src/pages/index/shared.tsx src/pages/index/shared.tsx
cp update/2026-07-21/src/pages/index/IndexMain.tsx src/pages/index/IndexMain.tsx
cp update/2026-07-21/src/pages/index/IndexTopbar.tsx src/pages/index/IndexTopbar.tsx
cp update/2026-07-21/src/pages/index/IndexSidebar.tsx src/pages/index/IndexSidebar.tsx
cp update/2026-07-21/src/pages/index/useDeepLinks.ts src/pages/index/useDeepLinks.ts
cp update/2026-07-21/src/pages/Index.tsx src/pages/Index.tsx

cp update/2026-07-21/src/pages/admin/FilesModal.tsx src/pages/admin/FilesModal.tsx
cp update/2026-07-21/src/pages/admin/adminShared.ts src/pages/admin/adminShared.ts
```

⚠️ Обратите внимание: раздел «Патчи» использует те же жёстко зашитые ссылки
на облачное S3-хранилище `cdn.poehali.dev` в качестве запасного варианта
(fallback), что и остальные функции с файлами. Если вы уже применяли патч
из `update/2026-07-18/README.md` (раздел про своё S3/MinIO) для
`backend/knowledge/index.py`, `backend/tasks/index.py` и
`backend/ideas/index.py` — точно так же нужно проверить переменную
`S3_PUBLIC_URL` и для `backend/patches/index.py`: если она задана в `.env`,
резервная ссылка на `cdn.poehali.dev` вообще не используется, ничего
патчить в коде не нужно.

### Шаг 2. Примените миграции базы данных

```bash
psql "$DATABASE_URL" -f update/2026-07-21/db_migrations/V0050__create_patch_files_table.sql
psql "$DATABASE_URL" -f update/2026-07-21/db_migrations/V0051__patch_files_task_ids_array.sql
psql "$DATABASE_URL" -f update/2026-07-21/db_migrations/V0052__create_patch_custom_roots.sql
psql "$DATABASE_URL" -f update/2026-07-21/db_migrations/V0053__seed_faq_patches_and_disk_usage.sql
```

⚠️ Если в вашей базе таблицы вынесены в отдельную схему (не `public`),
подключитесь с нужным `search_path` или впишите имя схемы в команды
миграций вручную перед запуском.

### Шаг 3. Установите Python-зависимости backend-функции

Функция `patches` использует `psycopg2-binary` и `boto3` — если эти
пакеты уже стоят у вас в общем venv backend (используются другими
функциями вроде `tasks`/`ideas`), отдельно ставить не нужно. Если у вас
venv на функцию — установите:

```bash
source venv/bin/activate   # или путь к вашему venv backend
pip install -r backend/patches/requirements.txt
```

### Шаг 4. Пропишите URL новой функции во фронтенде

Функция `patches` должна быть доступна на бэкенде так же, как остальные
функции (`tasks`, `ideas` и т.д.) — то есть через общий роутинг
`/api/patches`. Откройте `backend/func2url.json` и добавьте строку по
аналогии с уже существующими функциями:

```json
"patches": "/api/patches"
```

### Шаг 5. (Опционально) Точка проверки диска для статистики места

По умолчанию статистика места на диске (пункт 7) проверяет корень
файловой системы (`/`). Если хотите проверять конкретный раздел/точку
монтирования — добавьте в `.env`:

```
DISK_USAGE_PATH=/var/www/era
```

### Шаг 6. Проверьте лимит на размер запроса в Nginx

Каждый отдельный кусочек загрузки файла весит около 2 МБ (1.5 МБ данных +
запас на base64-кодирование и служебные поля JSON). Убедитесь, что в вашем
`deploy/nginx.conf` для блока `/api/` уже стоит достаточный
`client_max_body_size` (в проекте используется `420M` с запасом под другие
функции с вложениями) — если вы уже применяли фикс из
`update/2026-07-18/UPLOAD_FIX_SETUP.md`, ничего менять не нужно.

### Шаг 7. Пересоберите фронтенд и перезапустите backend

```bash
npm run build
sudo systemctl restart era-backend   # или: docker compose restart backend
```

## Проверка результата

1. Откройте сайт, в верхнем меню должен появиться новый пункт «Патчи».
2. Переключитесь между тремя серверами — дерево должно перезагружаться.
3. Перетащите тестовую папку с парой небольших файлов — должен появиться
   прогресс-бар, файлы должны появиться в дереве.
4. Создайте свою корневую папку кнопкой «+», убедитесь что удалить с
   файлами внутри нельзя, а пустую — можно.
5. Откройте любую задачу, выберите её в разделе «Патчи», кликните на
   скрепку у файла — файл должен подсветиться как прикреплённый.
6. Кликните на логотип «ЭРА» — должна открыться доска задач.
7. Откройте задачу/идею по прямой ссылке, нажмите «Доска» — адрес должен
   вернуться на `/`, обновление страницы должно показать доску, а не
   прежнюю карточку.
8. Откройте `Ядро → Файлы` в админке — должна быть видна полоса занятого
   места на диске.

## Если что-то пошло не так

- Пункт «Патчи» не появился в меню → убедитесь, что скопировали
  `IndexTopbar.tsx` и `IndexMain.tsx`, и что фронтенд пересобран.
- Дерево не грузится → проверьте, что применили все 3 миграции
  (`V0050`–`V0053`) и что в `func2url.json` прописан правильный
  адрес для ключа `patches`.
- Загрузка файла зависает или обрывается на середине → проверьте
  `client_max_body_size` в Nginx (см. Шаг 6) и логи backend-функции.
- Статистика диска показывает неверные цифры или ошибку → проверьте, что
  пользователь, от имени которого запущен backend-процесс, имеет права
  на чтение указанного пути (`DISK_USAGE_PATH` или `/` по умолчанию).

# Обновления от 21 июля

Один крупный новый раздел: «Патчи» — общее дерево файлов клиентского патча
по каждому серверу, с загрузкой перетаскиванием папки, привязкой файлов
к задачам и скачиванием архива по конкретной задаче. Раздел полностью новый —
ранее в проекте не существовал.

Краткое содержание:
1. Новый раздел «Патчи» — файловое дерево клиентского патча по серверам
2. Потоковая (chunked) загрузка файлов — обход лимита ~3 МБ на один запрос
3. Привязка/открепление уже загруженного файла к задаче прямо в дереве
4. Кнопка «Показать файлы патча этой задачи» в карточке задачи

## 1. Новый раздел «Патчи»

### Что это
Раздел показывает единое дерево файлов клиентского патча — отдельное
для каждого из трёх серверов (С4х1, HFx3 old, HF new). Дерево жёстко
организовано по 9 корневым папкам, которые соответствуют структуре клиента
игры: `animations`, `data`, `l2text`, `maps`, `staticmeshes`, `System`,
`System_eng`, `systextures`, `textures`.

Кто угодно из авторизованных участников может открыть раздел, посмотреть
дерево и скачать любой файл. Загружать, удалять файлы и привязывать их
к задачам могут только администраторы и участники с правом полного
редактирования задач (`task_edit_own`).

### Как загружать файлы
1. Откройте раздел «Патчи» в верхнем меню, выберите нужный сервер.
2. При необходимости выберите задачу в выпадающем списке над деревом —
   тогда все файлы, загруженные ниже, автоматически прикрепятся к ней.
3. Перетащите папку (например «System» целиком или только «data») прямо
   на нужную корневую папку в дереве — вложенная структура сохранится
   один в один.
4. Во время загрузки виден прогресс-бар с процентом и номером текущего
   файла из очереди, загрузку можно отменить кнопкой «Отменить».

### Привязка уже загруженного файла к задаче
Если файл уже лежит в дереве, а прикрепить его к задаче нужно позже:
выберите задачу в выпадающем списке над деревом, наведите курсор на
нужный файл — справа появится иконка скрепки. Клик прикрепляет файл
к выбранной задаче, повторный клик — открепляет. Один файл может быть
одновременно прикреплён к нескольким задачам — привязки не заменяют,
а дополняют друг друга.

⚠️ Важный нюанс: если файл по одному и тому же пути загружается повторно
(новая версия поверх старой), в дереве остаётся только последняя версия
файла — сервер не хранит историю версий. При этом все прежние привязки
к задачам за файлом сохраняются, плюс добавляется новая. То есть если
файл был прикреплён к задаче №1, а затем перезалит с меткой задачи №2 —
он будет одновременно числиться и на задаче №1, и на задаче №2, но
содержимое файла на обеих задачах будет уже новым (с последними правками).

### Скачивание файлов задачи
Если в выпадающем списке выбрана задача, рядом появляется кнопка
«Скачать файлы задачи (N)» — она собирает все прикреплённые к этой
задаче файлы в один zip-архив.

### Файлы для переноса
```
backend/patches/index.py
backend/patches/requirements.txt
backend/patches/tests.json
src/pages/index/Patches.tsx
src/pages/index/shared.tsx
src/pages/index/IndexMain.tsx
src/pages/index/IndexTopbar.tsx
src/pages/Index.tsx
db_migrations/V0050__create_patch_files_table.sql
db_migrations/V0051__patch_files_task_ids_array.sql
```

### Проверка
- Откройте раздел «Патчи», переключите сервер — дерево должно
  перезагрузиться, показав 9 фиксированных корневых папок.
- Перетащите тестовую папку (например с парой файлов внутри) на любую
  корневую папку — должен появиться прогресс-бар, а по завершении файлы
  должны появиться в дереве с правильной вложенностью.
- Выберите задачу в списке, наведите на файл — должна появиться иконка
  скрепки; кликните — файл должен подсветиться, повторный клик — подсветка
  должна пропасть.
- Откройте любую задачу — внизу должна быть кнопка «Показать файлы патча
  этой задачи», по клику должен открыться раздел «Патчи» с уже выбранной
  этой задачей и подсвеченными прикреплёнными файлами.

## 2. Потоковая (chunked) загрузка — обход лимита на размер запроса

### Что было не так
Первая версия раздела отправляла содержимое файла одним запросом целиком
(в виде base64 в теле JSON). На практике оказалось, что инфраструктура
физически режет тело одного запроса к серверной функции на уровне
~3 МБ — то есть большинство настоящих файлов клиентского патча (текстуры,
модели, скомпилированные `.u`/`.dll` файлы) вообще не могли загрузиться:
запрос либо отклонялся с ошибкой 413, либо функция падала без ответа
(«Failed to fetch»/503) при чуть большем объёме.

### Что исправлено
Каждый файл теперь автоматически режется на стороне браузера на кусочки
по 1.5 МБ и отправляется последовательно отдельными запросами:
1. `file_init` — сообщает серверу путь файла и (если выбрана) задачу,
   получает временный `fileId`.
2. `file_chunk` — грузит по очереди каждый кусочек (используя `fileId`
   и номер по порядку), кусочки временно складываются в S3.
3. `file_complete` — сервер склеивает все кусочки в один файл (до 200 МБ),
   сохраняет его в постоянном месте и записывает в базу.
4. `file_abort` — если загрузка отменена или произошла ошибка, все
   временные кусочки этого файла удаляются из S3.

Благодаря этому загружаются файлы любого реального размера клиентского
патча (проверено на файле 30 МБ), а не только совсем маленькие.

### Файлы для переноса
```
backend/patches/index.py
src/pages/index/Patches.tsx
```

### Проверка
- Загрузите через перетаскивание файл размером больше 5 МБ (раньше такой
  не проходил) — должен загрузиться полностью, с корректным прогресс-баром.
- Загрузите папку с несколькими файлами разного размера одновременно —
  все должны загрузиться по очереди без ошибок.
- Нажмите «Отменить» посреди загрузки — процесс должен остановиться,
  частично загруженные кусочки не должны остаться висеть в файловом
  хранилище (проверяется автоматически через `file_abort`).

## Как перенести оба изменения к себе на сервер

Раздел «Патчи» полностью новый — специальных инструкций по настройке
cron или systemd не требуется, только применение миграций и копирование
файлов, описанное ниже.

### Шаг 1. Скопируйте новые файлы

```bash
cd /var/www/era   # корень вашего проекта на сервере

mkdir -p backend/patches
cp update/2026-07-21/backend/patches/index.py backend/patches/index.py
cp update/2026-07-21/backend/patches/requirements.txt backend/patches/requirements.txt
cp update/2026-07-21/backend/patches/tests.json backend/patches/tests.json

cp update/2026-07-21/src/pages/index/Patches.tsx src/pages/index/Patches.tsx
cp update/2026-07-21/src/pages/index/shared.tsx src/pages/index/shared.tsx
cp update/2026-07-21/src/pages/index/IndexMain.tsx src/pages/index/IndexMain.tsx
cp update/2026-07-21/src/pages/index/IndexTopbar.tsx src/pages/index/IndexTopbar.tsx
cp update/2026-07-21/src/pages/Index.tsx src/pages/Index.tsx
```

### Шаг 2. Примените миграции базы данных

```bash
psql "$DATABASE_URL" -f update/2026-07-21/db_migrations/V0050__create_patch_files_table.sql
psql "$DATABASE_URL" -f update/2026-07-21/db_migrations/V0051__patch_files_task_ids_array.sql
```

⚠️ Если в вашей базе таблицы вынесены в отдельную схему (не `public`),
подключитесь с нужным `search_path` или впишите имя схемы в команды
миграций вручную перед запуском.

### Шаг 3. Установите Python-зависимости backend-функции

Функция `patches` использует `psycopg2-binary` и `boto3` — если эти
пакеты уже стоят у вас в общем venv backend (используются другими
функциями вроде `tasks`/`ideas`), отдельно ставить не нужно. Если у вас
venv на функцию — установите:

```bash
source venv/bin/activate   # или путь к вашему venv backend
pip install -r backend/patches/requirements.txt
```

### Шаг 4. Пропишите URL новой функции во фронтенде

Функция `patches` должна быть доступна на бэкенде так же, как остальные
функции (`tasks`, `ideas` и т.д.) — то есть через общий роутинг
`/api/patches` в вашем `deploy/server.py`/Nginx-конфиге. Откройте
`backend/func2url.json` и добавьте строку по аналогии с уже существующими
функциями (посмотрите, как оформлена соседняя строка `"tasks"` в этом же
файле, и оформите `"patches"` точно так же):

```json
"patches": "/api/patches"
```

### Шаг 5. Проверьте переменные окружения S3

Функция `patches` хранит файлы в том же S3/MinIO хранилище, что и
вложения к задачам — использует те же переменные окружения:
`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `S3_BUCKET` (по умолчанию
`files`), `S3_ENDPOINT` (по умолчанию `https://bucket.poehali.dev` —
если у вас свой MinIO, впишите свой адрес), `S3_PUBLIC_URL` (публичный
CDN-адрес для скачивания файлов). Если вложения к задачам у вас уже
работают — значит эти переменные уже настроены и трогать их не нужно.

### Шаг 6. Проверьте лимит на размер запроса в Nginx

Даже с новой потоковой загрузкой каждый отдельный кусочек весит около
2 МБ (1.5 МБ данных + запас на base64-кодирование и служебные поля JSON).
Убедитесь, что в вашем `deploy/nginx.conf` для блока `/api/` уже стоит
достаточный `client_max_body_size` (в проекте используется `420M` с
запасом под другие функции с вложениями) — если вы уже применяли фикс
из `update/2026-07-18/UPLOAD_FIX_SETUP.md`, ничего менять не нужно.

### Шаг 7. Пересоберите фронтенд и перезапустите backend

```bash
npm run build
sudo systemctl restart era-backend   # или: docker compose restart backend
```

## Проверка результата

1. Откройте сайт, в верхнем меню должен появиться новый пункт «Патчи».
2. Переключитесь между тремя серверами — дерево должно перезагружаться
   и показывать одни и те же 9 корневых папок для каждого сервера.
3. Перетащите тестовую папку с парой небольших файлов на любую корневую
   папку — должен появиться прогресс-бар, файлы должны появиться в дереве.
4. Откройте любую задачу, выберите её в списке в разделе «Патчи», кликните
   на скрепку у файла — файл должен подсветиться как прикреплённый к задаче.
5. Скачайте архив файлов задачи кнопкой «Скачать файлы задачи» — должен
   скачаться zip с правильными файлами.

## Если что-то пошло не так

- Пункт «Патчи» не появился в меню → убедитесь, что скопировали
  `IndexTopbar.tsx` и `IndexMain.tsx`, и что фронтенд пересобран
  (`npm run build`) после копирования файлов.
- Дерево не грузится (пустой экран/крутится загрузка) → проверьте, что
  применили обе миграции (`V0050`, `V0051`) и что в `func2url.json`
  прописан правильный адрес для ключа `patches`.
- Загрузка файла зависает или обрывается на середине → проверьте
  `client_max_body_size` в Nginx (см. Шаг 6) и логи backend-функции
  (`sudo journalctl -u era-backend -n 100 --no-pager` или
  `docker compose logs -f backend`) — там будет видна конкретная ошибка
  S3/базы.
- Файлы загружаются, но не видны на CDN-ссылке → проверьте переменную
  `S3_PUBLIC_URL`/`S3_ENDPOINT` — она должна указывать на реально
  доступный извне адрес вашего S3/MinIO.

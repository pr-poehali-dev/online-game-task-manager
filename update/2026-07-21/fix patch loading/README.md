# Фикс: загрузка патчей + положение скрепки в комментариях

Эта папка — дополнение к сегодняшнему обновлению (`update/2026-07-21`).
Она содержит два отдельных исправления, которые не попали в основной апдейт вовремя:

1. **Критичный фикс загрузки файлов** в разделе «Патчи» (и заодно во всех
   остальных местах, где загружаются файлы — вложения к задачам, идеям,
   статьям базы знаний, файлы в админке).
2. **Фикс положения иконки-скрепки** (прикрепление файла) в комментариях
   к задачам и идеям — она была случайно пропущена в предыдущем апдейте.

---

## 1. Почему не грузились патчи

### Что было не так
Обнаружены две связанные проблемы при загрузке файлов:

1. Файлы размером **0 байт** (частое явление в реальных клиентских
   патчах — пустые placeholder-файлы) отклонялись с ошибкой 400, потому
   что код проверял содержимое кусочка на пустоту (`if not data_b64`), а
   пустой файл после кодирования в base64 как раз и даёт пустую строку.
2. Более серьёзная проблема: новые версии библиотеки `botocore` по
   умолчанию добавляют к запросу контрольную сумму через специальный
   chunked-трейлер. Кастомное (не-AWS) S3-хранилище иногда не разбирает
   этот трейлер как надо — в результате в конец сохранённого файла
   попадал технический «мусор» (например строка вида
   `x-amz-checksum-crc32:...`), из-за чего файл оказывался повреждён,
   даже если запрос возвращал успешный ответ.

### Что исправлено
- Проверка на пустой кусочек данных заменена на явную проверку `is None`
  вместо проверки на пустое значение — 0-байтные файлы теперь
  загружаются корректно.
- В клиент S3 добавлена конфигурация `botocore.config.Config` с
  отключением автоматического расчёта/проверки контрольных сумм
  (`request_checksum_calculation='when_required'`,
  `response_checksum_validation='when_required'`) — файлы теперь
  сохраняются байт-в-байт, без постороннего мусора в конце.

Фикс применён во всех backend-функциях, которые работают с файлами:
`patches`, `tasks`, `ideas`, `knowledge`, `admin`.

### ⚠️ Важно: нужна актуальная версия `boto3` на вашем сервере
Параметры `request_checksum_calculation` / `response_checksum_validation`
появились в библиотеке `botocore` только начиная с версии **1.36**, которая
подтягивается вместе с `boto3` версии **1.36 и выше**. Если на вашем
сервере ранее уже было установлено Python-окружение со старой версией
`boto3` (например если раньше в `requirements.txt` стояло просто
`boto3` без указания минимальной версии, или зависимости ставились давно
и не обновлялись) — старая версия `botocore` **не знает** про эти два
параметра, и попытка их использовать приведёт к ошибке `TypeError`,
которая полностью ломает загрузку файлов (именно поэтому у вас файлы не
заливались, хотя тут, на poehali.dev, где `boto3` всегда актуальный,
всё работало нормально).

Сам код на этот случай теперь защищён (обёрнут в `try/except`), но чтобы
получить настоящую пользу от фикса (без мусора в конце файлов),
обновите библиотеку на сервере:
```bash
cd /var/www/era
./venv/bin/pip install --upgrade "boto3>=1.36.0"
sudo systemctl restart era-backend
```
В `deploy/requirements.txt` требование тоже обновлено на `boto3>=1.36.0`.

### Файлы для переноса
```
backend/patches/index.py
backend/tasks/index.py
backend/ideas/index.py
backend/knowledge/index.py
backend/admin/index.py
```

Все пять файлов уже адаптированы под своё S3/MinIO-хранилище (без
жёстко зашитого облачного адреса poehali.dev) — адрес хранилища и
публичный адрес файлов берутся из переменных `S3_ENDPOINT`,
`S3_PUBLIC_URL` (при необходимости `CDN_BASE_URL`), с запасным
вариантом на локальный MinIO по адресу `http://127.0.0.1:9000`.
Убедитесь, что в `.env` эти переменные заданы под ваш сервер.

⚠️ Если вы уже применяли предыдущий апдейт (`update/2026-07-21/README.md`)
и меняли в нём `backend/patches/index.py` / `backend/admin/index.py` —
файлы отсюда нужно скопировать **поверх** них, они содержат все прошлые
изменения плюс этот новый фикс.

### Проверка после переноса
- Загрузите пустой файл 0 байт (в «Патчах» или как вложение к
  задаче/идее/статье) — должен загрузиться без ошибок.
- Загрузите обычный файл с текстовым содержимым, скачайте его обратно и
  сравните — размер и содержимое должны совпадать один в один, без
  лишних байт в конце.

## 2. Положение скрепки (прикрепление файла) в комментариях

### Что было не так
В предыдущем апдейте иконка-скрепка для прикрепления файла к
комментарию была расположена **внутри** поля ввода текста — из-за этого
она визуально «путалась» с текстом и была неудобна при наборе длинных
комментариев.

### Что исправлено
Иконка-скрепка вынесена **наружу**, в отдельный вертикальный столбик
рядом с кнопкой отправки (Send) — теперь это отдельная кнопка с
иконкой Paperclip, а список уже прикреплённых файлов отображается
компактными «чипами» под полем ввода.

### Файлы для переноса
```
src/components/AttachmentsField.tsx
src/pages/index/TaskComments.tsx
src/pages/index/ideas/IdeaComments.tsx
```

### Проверка после переноса
- Откройте любую задачу или идею, прокрутите к комментариям.
- Скрепка должна быть отдельной кнопкой справа от поля ввода (над
  кнопкой отправки), а не внутри текстового поля.
- Прикрепите файл к комментарию — он должен появиться компактным
  «чипом» под полем ввода с возможностью удалить его до отправки.
